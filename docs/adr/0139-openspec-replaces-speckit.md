# ADR 0139 — OpenSpec replaces SpecKit

**Status:** Accepted

## Context

ADR 0056 introduced SpecKit (github/spec-kit) as the built-in spec-driven development framework. SpecKit implements a 7-phase sequential pipeline:

1. `speckit.constitution` — project principles and constraints
2. `speckit.specify` — user stories and acceptance criteria
3. `speckit.plan` — technical architecture
4. `speckit.tasks` — task decomposition
5. `speckit.implement` — implementation
6. `speckit.clarify` — ambiguity resolution
7. `speckit.analyze` — drift detection

In practice this pipeline has two friction points:

- **Token cost.** Each phase produces 800–1 200 lines of structured output. Carrying seven phases of history into downstream turns is the primary driver of context bloat for spec-driven sessions, even with the SpecSlicer optimisation from ADR 0100.
- **Brownfield mismatch.** SpecKit was designed greenfield-first. Adopting it mid-project requires custom workarounds; the constitution phase is redundant when a standing `CONSTITUTION.md` already exists.

The project's constitution (`docs/spec/CONSTITUTION.md`) is now a stable standing artefact maintained independently of any spec command.

## Decision

Replace SpecKit with **OpenSpec** (Fission-AI/OpenSpec) as the built-in framework. OpenSpec compresses the workflow to three commands and is brownfield-first by design.

### New commands

| Command | Purpose |
|---|---|
| `openspec.propose` | Elicit requirements, write `proposal.md`, `design.md`, `tasks.md` |
| `openspec.review` | Audit the three artefacts; produce gap report before implementation |
| `openspec.apply` | Execute tasks in order; archive completed change to `openspec/specs/` |

### File layout

```
openspec/
  changes/<change-name>/
    proposal.md      ← requirements, user stories, acceptance criteria
    design.md        ← tech stack, architecture, data model, security
    tasks.md         ← ordered T-01 … T-NN implementation checklist
  specs/             ← living library; changes are archived here after apply
    INDEX.md         ← one-row-per-change summary table
```

### Decisions

**Full replacement, not parallel support.** SpecKit is removed entirely to reduce code surface. The config value `"spec-kit"` is no longer recognised and will fall through to the directory-path lookup, log a warning, and disable the framework.

**Dot-namespaced command names.** The upstream OpenSpec project uses an `opsx:` colon prefix. We adopt dot-namespacing (`openspec.propose`) to match the existing slash-command parser convention and slash-menu autocomplete UX. No parser changes are required.

**All three commands clear context.** Consistent with ADR 0097 policy: every spec-framework phase command is a discrete workflow entry point that starts with a clean context window. All three OpenSpec commands set `clears_context: true`.

**SpecSlicer extended for OpenSpec paths (ADR 0100 extension).** `SpecSlicer::build_openspec(change_dir)` reads from `<change_dir>/tasks.md` and `<change_dir>/design.md` rather than the SpecKit paths `TASKS.md` / `SPEC.md`. The parsing logic (`parse_active_task`, `slice_spec`) is unchanged — it operates on any markdown. Injection is triggered on `openspec.apply` only (parallel to the former `speckit.implement` trigger).

**Stable numbering conventions retained (ADR 0103).** OpenSpec templates adopt the same identifier schemes: `US-001` for stories, `T-01` for tasks, `G-01` / `R-01` / `S-01` for the review gap report.

**Constitution is a shared standing artefact.** `docs/spec/CONSTITUTION.md` is not generated by any OpenSpec command. The `openspec.propose` template reads it as a hard constraint. Users update it manually or outside of OpenSpec.

### What is NOT migrated

Existing SpecKit output under `docs/spec/features/*/` is not touched. Those files remain on disk as historical artefacts. New work uses `openspec/changes/<name>/`. There is no collision between the two directory trees.

## Consequences

- **Token savings per apply session.** Three-command flow with ~250 lines of output vs. seven-phase flow with ~800 lines reduces per-session context pressure.
- **Simpler slash-menu.** Three commands instead of seven — less cognitive overhead when deciding which phase to run.
- **Brownfield adoption.** `openspec.propose` can be run against any existing codebase without bootstrapping a constitution or prior spec artefacts.
- **No backward compatibility for `"spec-kit"` config value.** Users must update `config.toml` to `spec_framework = "open-spec"`.
